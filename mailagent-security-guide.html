Authentication — from the very beginning

Authentication answers one question: who are you? Authorization answers a second: what are you allowed to do? Every security system separates these two steps — even though they're often confused.

When you visit a website and log in, the server has a problem: HTTP is stateless. Each request is completely independent. After you prove who you are in request #1, request #2 has no memory of that. The server has to figure out how to remember that you authenticated, across every future request.

The solution to this is either a session (the server remembers you) or a token (you carry proof of who you are with every request). Both approaches have tradeoffs.

Passwords — how they actually work

The wrong way (and why it's catastrophic)

The most dangerous thing you can do is store passwords in plain text. If your database is ever leaked — through an SQL injection, a backup misconfiguration, a disgruntled employee — every user's password is immediately readable. Because people reuse passwords across sites, this also compromises their Gmail, their bank, everything.

Encrypting passwords is also wrong. Encryption is reversible — if you can decrypt, so can an attacker who gets your encryption key.

The right way: cryptographic hashing

A hash function is a one-way mathematical transformation. You put in a password, you get out a fixed-length string. You cannot reverse it. SHA-256("hello") always produces the same output, but you can never go from the output back to "hello" — you can only check if a given input produces that output.

But plain SHA-256 on a password is still wrong. Here's why:

Attack: Rainbow Table ───────────────────────────────────────────────── An attacker pre-computes SHA-256 of every common password: SHA-256("password123") = "ef92b..." SHA-256("hello") = "2cf24..." SHA-256("qwerty") = "65e84..." (billions of entries) They steal your DB and find a hash "ef92b..." — lookup in their table instantly reveals the plaintext was "password123". Counter-attack: Salting ───────────────────────────────────────────────── Before hashing, append a random string (the "salt") unique per user: HASH("password123" + "kx7q9r") → completely different output HASH("password123" + "m2p8w3") → different again Rainbow tables become useless — each user's salt makes their hash unique.

bcrypt, scrypt, argon2 — the correct tools

These are password-specific hashing algorithms designed to be intentionally slow. That sounds wrong, but it's crucial. A fast hash (SHA-256) can be computed 10 billion times per second on GPU hardware. bcrypt, by design, takes 100ms. Cracking 1 million leaked passwords with SHA-256: hours. With bcrypt: centuries.

AlgorithmSpeedMemory-hard?Recommendation
SHA-256 (wrong)~10B/sec GPUNoNever for passwords
MD5 (wrong)~50B/sec GPUNoNever
bcrypt~100msNoGood default
scrypt~100msYesBetter than bcrypt
argon2id~100msYesBest current choice

"Memory-hard" means the algorithm requires large amounts of RAM, making GPU-based cracking expensive even if time is available.

Sessions vs JWT — how servers remember you

Session-based auth (server-side)

Login flow: 1. User sends credentials 2. Server verifies, creates a session record in DB: {session_id: "abc123", user_id: "u1", expires: ...} 3. Server sends back a cookie: Set-Cookie: session_id=abc123; HttpOnly; Secure Every subsequent request: 1. Browser automatically sends Cookie: session_id=abc123 2. Server looks up "abc123" in session store → gets user_id → authorized 3. If session_id not found or expired → reject Logout: Delete the session record from DB → immediately invalid

Advantages: Instant revocation (delete from DB). Server has full control. Session data lives server-side.
Disadvantages: Requires DB lookup on every request. Doesn't scale horizontally without shared session store (Redis). Stateful.

JWT — JSON Web Tokens (client-side)

JWT structure: header.payload.signature header: {"alg": "HS256", "typ": "JWT"} payload: {"user_id": "u1", "email": "dev@x.com", "exp": 1700000000} signature: HMAC-SHA256(base64(header) + "." + base64(payload), SECRET_KEY) Login flow: 1. User sends credentials 2. Server verifies, creates JWT and sends it back 3. Client stores JWT (localStorage or cookie) Every subsequent request: 1. Client sends: Authorization: Bearer eyJhbGc... 2. Server verifies signature (no DB lookup needed) 3. Checks expiry in payload 4. Reads user_id directly from payload — that's the logged-in user

Advantages: Stateless — no DB lookup per request. Scales horizontally. Self-contained.
Disadvantages: Cannot be instantly revoked (token is valid until expiry). Logout is client-side only (deleting the token). If a token is stolen, it's valid until it expires. Payload is base64-encoded, NOT encrypted — anyone can read it (never put secrets in payload).

JWT misconception

JWT is NOT encryption. atob(payload) in any browser decodes the payload completely. The signature only proves the payload hasn't been tampered with. Use short expiry times (15 minutes) and refresh tokens for longer sessions.

OAuth 2.0 / Google Login — the complete flow

OAuth solves a specific problem: you want to let a third-party app (your MailingAgent) access a user's resources (their Gmail) without the user giving you their Google password. You'd have to store it, secure it, and become responsible for it. Instead, OAuth lets the user grant you access through Google directly.

The OAuth 2.0 Authorization Code Flow (what your app uses): 1. USER CLICKS "Login with Google" Your server generates: - state = random 32-byte string (CSRF protection) - Store state in user's session Redirect user to Google: https://accounts.google.com/o/oauth2/auth ?client_id=YOUR_CLIENT_ID &redirect_uri=https://yourapp.com/auth/callback &scope=gmail.readonly calendar.events &response_type=code &access_type=offline &state=abc123xyz ← critical CSRF token 2. USER SEES GOOGLE'S CONSENT SCREEN "MailingAgent wants to read your Gmail and access your Calendar" User clicks Allow. 3. GOOGLE REDIRECTS BACK TO YOUR CALLBACK GET /auth/callback?code=4/P7q...&state=abc123xyz Your server: a. Verifies state matches what you stored — prevents CSRF b. Exchanges code for tokens: POST https://oauth2.googleapis.com/token code=4/P7q... client_id=... client_secret=... grant_type=authorization_code c. Google responds: { "access_token": "ya29.a...", ← use to call Gmail/Calendar API "refresh_token": "1//0g...", ← use to get new access tokens "expires_in": 3600 ← access token lasts 1 hour } d. Encrypt and store tokens in oauth_credentials table e. Create your own session/JWT for the user 4. SUBSEQUENT API CALLS Your backend decrypts access_token from DB and calls Gmail API When access_token expires, use refresh_token to get a new one (without bothering the user again)
The state parameter — why it's not optional

Without the state parameter, an attacker can craft a Google OAuth callback URL and trick a victim's browser into executing it. This lets the attacker link their own Google account to the victim's account — hijacking it. This is a CSRF attack on the OAuth flow, and it's one of the critical vulnerabilities in your current code.

OTP Login (One-Time Passwords)

OTP login sends a temporary code (usually 6 digits) via SMS or email. The user enters it to authenticate. It's passwordless — the phone number or email address IS the identity.

Flow: 1. User enters phone number / email 2. Server generates 6-digit code, hashes it, stores in DB with expiry 3. Sends code via SMS (Twilio) or email (SendGrid) 4. User enters code 5. Server verifies: HASH(entered_code) == stored_hash AND not expired 6. Code marked as used (single-use) 7. Create session/JWT Key security details: - Store a HASH of the code, not the plaintext (same principle as passwords) - Short expiry: 10 minutes maximum - Single use: once verified, invalidate immediately - Rate limit: maximum 3 attempts before lockout - Rate limit sending: maximum 3 SMS per phone per hour (prevents SMS bombing) - Timing-safe comparison: use hmac.compare_digest(), not ==

Forgot Password — how it works securely

Secure forgot-password flow: 1. User requests reset for email@example.com Server response: "If that email exists, we've sent a reset link" (Never reveal whether an email is registered — prevents user enumeration) 2. Server: - Generates a cryptographically random token: secrets.token_urlsafe(32) - Hashes the token: SHA-256(token) - Stores the HASH in DB with: user_id, expires_at (1 hour), used=false - Emails the UNHASHED token: https://app.com/reset?token=abc123... 3. User clicks link Server: - Takes the token from URL - Hashes it: SHA-256(token) - Looks up that hash in DB — must exist, not expired, not used - Shows password reset form 4. User submits new password Server: - Marks token as used (single-use) - Hashes new password with argon2id - Updates user record - Invalidates all existing sessions for that user
Why store the hash of the reset token?

Same reason you hash passwords. If the reset_tokens table is leaked, attackers can't use the stored values to take over accounts. The token in the URL is the secret; the hash in the DB is useless without it.

1 — Authentication and Authorization

These are two different things that work together.

Authentication (AuthN) — proving identity. "I am Devanshu." Verified by a password, a Google login, a biometric, or a token.

Authorization (AuthZ) — proving permission. "Devanshu is allowed to read conversation #42." Verified by checking a policy, a role, or an ownership record.

Most security vulnerabilities happen when authorization is missing — the system authenticates the user but then forgets to check whether that user is allowed to access the specific resource they're asking for.

Dependency injection for auth in FastAPI (the right pattern)

        raise HTTPException(status_code=401, detail="Invalid token")

Input validation is the principle that you never trust data that comes from outside your system. Every piece of user-supplied data — request bodies, URL parameters, query strings, headers — is potentially malicious. You validate before processing.

3 — Secret Keys and Environment Variables

Secrets are values that must remain private — API keys, database passwords, signing keys, encryption keys. The core principle is: secrets must never appear in your codebase, in your logs, in error messages, or in client-facing responses.

How environment variables work

Development: .env file → Pydantic Settings reads it → settings.anthropic_api_key Production: AWS Secrets Manager or Parameter Store → injected as environment variable when ECS container starts → Pydantic Settings reads process environment → never written to disk on the container The .env file must be in .gitignore. It must never be committed. If it is ever committed, assume the keys are compromised — rotate them.

Key management rules

RuleWhy
One key, one purposeA key used for encryption should not also be used for signing. Reuse creates coupling — rotating one key breaks the other.
Minimum key length32 bytes minimum for symmetric keys. 256-bit RSA is breakable — use 2048-bit minimum.
Keys must be randomNever derive a key from a password or a phrase. Use secrets.token_bytes(32) or a proper key derivation function.
Rotation planEvery secret must have a documented rotation procedure. You must be able to rotate a key without downtime.
Never log secretsCheck your logging configuration. Pydantic Settings can accidentally log all settings on startup.
Never expose in errorsA 500 error that includes the connection string leaks your DB password.
Separate per environmentDev DB, staging DB, and prod DB must have completely different credentials.

4 — SQL Injection

SQL injection is the most common and destructive web vulnerability. It happens when user-supplied data is embedded directly into SQL strings without sanitization, allowing attackers to break out of the intended query and run arbitrary SQL.

Vulnerable code: user_input = "'; DROP TABLE users; --" query = f"SELECT * FROM users WHERE email = '{user_input}'" Becomes: SELECT * FROM users WHERE email = ''; DROP TABLE users; --' Result: your entire users table is deleted. Variant — data exfiltration: user_input = "' OR '1'='1" query = f"SELECT * FROM users WHERE email = '{user_input}'" Becomes: SELECT * FROM users WHERE email = '' OR '1'='1' Result: returns ALL users — authentication bypassed.

The defense: parameterized queries (always)

# WRONG — string formatting
query = f"SELECT * FROM users WHERE email = '{user_email}'"
await db.fetch(query)

# WRONG — string concatenation
query = "SELECT * FROM users WHERE email = '" + user_email + "'"

# RIGHT — parameterized (asyncpg)
await db.fetch("SELECT * FROM users WHERE email = $1", user_email)

# RIGHT — SQLAlchemy ORM
result = await session.execute(
    select(User).where(User.email == user_email)
)

# RIGHT — parameterized (psycopg2 / sync)
cursor.execute("SELECT * FROM users WHERE email = %s", (user_email,))

With parameterized queries, the database treats $1 as a value slot — the user's input can never become part of the SQL structure. It's always just data.

Second-order injection

More subtle: data stored safely in the DB is later retrieved and embedded unsafely into a new query. Example: you safely insert a username of admin'-- into the users table. Later, a reporting query retrieves that name and uses it unsafely in a generated query. The defense: parameterize every query, even ones using data from your own database.

ORM isn't automatic protection

# This is safe:
User.query.filter_by(email=user_email).first()

# This is NOT safe — raw SQL in ORM:
db.execute(f"SELECT * FROM users WHERE email = '{user_email}'")

5 — CORS and Security Headers

CORS — Cross-Origin Resource Sharing

Browsers enforce a same-origin policy: JavaScript on https://evil.com cannot make requests to https://yourapi.com and read the response. Without this, any website could make API calls on behalf of a logged-in user and steal their data.

CORS is the mechanism that lets servers explicitly say "I allow requests from these specific origins." It works through headers.

Browser preflight check (for POST/PUT/DELETE requests): OPTIONS /chat/123/message HTTP/1.1 Origin: https://yourfrontend.com Access-Control-Request-Method: POST Server response: Access-Control-Allow-Origin: https://yourfrontend.com Access-Control-Allow-Methods: GET, POST, OPTIONS Access-Control-Allow-Headers: Authorization, Content-Type Access-Control-Allow-Credentials: true ← needed for cookies/auth headers Browser sees "this origin is allowed" → sends the real request
The most common CORS mistake

Setting Access-Control-Allow-Origin: * (allow all origins) in combination with Access-Control-Allow-Credentials: true is invalid and browsers reject it. But setting Allow-Origin: * without credentials means any website can read your API responses — fine only if all data is public.

FastAPI CORS configuration (correct pattern)

from fastapi.middleware.cors import CORSMiddleware

ALLOWED_ORIGINS = [
    "http://localhost:5173",          # Vite dev server
    "https://yourapp.com",            # Production frontend
    "app://.",                        # Electron app origin
]

app.add_middleware(
    CORSMiddleware,
    allow_origins=ALLOWED_ORIGINS,    # Never use ["*"] with credentials
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["Authorization", "Content-Type"],
)

6 — Rate Limiting and Abuse Protection

Rate limiting restricts how many requests a caller can make in a given time window. Without it, any endpoint is vulnerable to brute force, abuse, and denial-of-service.

What to rate limit

EndpointWhySuggested limit
Auth endpoints (/auth, /login)Brute force prevention5 per IP per 15 minutes
LLM-invoking endpoints (/chat)LLM calls cost money; abuse can bankrupt you20 per user per hour
Send email endpointsYour app can send on behalf of users — abuse causes real harm30 per user per hour (yours)
OTP/password resetPrevents SMS bombing and reset token exhaustion3 per email per hour
Approval endpointsPrevents replay attacks and approval flooding10 per user per minute

The sliding window approach with Redis

Simple fixed window (WRONG for auth): "5 requests per minute" Problem: at 11:59pm you use 5. At 12:00am the window resets. At 12:00:01am you use 5 more = 10 requests in 2 seconds. Sliding window (correct): At any given moment, count requests in the last 60 seconds. Uses Redis sorted sets with timestamps as scores.
import redis.asyncio as redis
import time

r = redis.from_url(settings.redis_url)

async def rate_limit(key: str, limit: int, window_seconds: int) -> bool:
    now = time.time()
    pipe = r.pipeline()
    pipe.zremrangebyscore(key, 0, now - window_seconds)  # remove old entries
    pipe.zadd(key, {str(now): now})                       # add current request
    pipe.zcard(key)                                       # count in window
    pipe.expire(key, window_seconds)                      # TTL cleanup
    results = await pipe.execute()
    count = results[2]
    return count <= limit

# Usage in endpoint
@router.post("/{conversation_id}/message")
async def send_message(
    conversation_id: str,
    instruction: str,
    current_user_id: str = Depends(get_current_user)
):
    if not await rate_limit(f"chat:{current_user_id}", limit=20, window_seconds=3600):
        raise HTTPException(status_code=429, detail="Too many requests")
    ...

Atomic rate limit check — the race condition problem

Your current rate limit is broken

Your rate_limit.py queries the audit_log: SELECT count(*) ... WHERE created_at > now() - interval '1 hour'. Then if count < 30, it allows the request. Then the action happens. Then the audit log is written. Between the check and the write, 50 concurrent requests can all pass the check simultaneously. The Redis pipeline approach above is atomic — the check and increment happen in one operation.

Exponential backoff — punishing repeated failures

async def get_lockout_delay(user_id: str) -> float:
    failures = await r.get(f"auth_failures:{user_id}") or 0
    failures = int(failures)
    if failures >= 10:
        raise HTTPException(status_code=429, detail="Account temporarily locked")
    return min(2 ** failures, 30)  # 1s, 2s, 4s, 8s, 16s, 30s, 30s...

7 — SSL / TLS

SSL (now technically TLS — SSL was deprecated) provides encryption in transit. It ensures that traffic between the browser and your server cannot be read or tampered with by anyone in between (your ISP, a coffee shop's WiFi router, a government, etc.).

How TLS works (simplified)

TLS Handshake: 1. Client → Server: "I support these cipher suites: [TLS_AES_256_GCM, ...]" 2. Server → Client: "Here is my certificate (signed by a trusted CA like Let's Encrypt)" 3. Client verifies the certificate: - Is it signed by a CA my browser trusts? - Does the domain match? - Is it expired? 4. They agree on an encryption key using asymmetric crypto (RSA/ECDH) (The key exchange itself can't be decrypted even if recorded) 5. All further communication uses symmetric encryption with that key Result: - Data is encrypted (confidentiality) - Data can't be tampered with (integrity) - Server identity is verified (authentication)

In production — where TLS lives

Option A: CloudFront + ALB (your architecture) Browser → CloudFront (HTTPS) → ALB (HTTPS) → ECS Container (HTTP internally) TLS is terminated at CloudFront/ALB. Your container speaks plain HTTP inside the VPC. This is correct — the internal VPC network is private and trusted. Option B: Nginx termination Browser → Nginx (HTTPS with Let's Encrypt cert) → App (localhost HTTP) Option C: Direct container (wrong for production) Running uvicorn with --ssl-keyfile and --ssl-certfile directly This works but makes cert rotation complex and bypasses CDN benefits.

Code scan — Secrets and Environment Variables

CriticalReal Fernet key exposed in README.md

In the README configuration table:

TOKEN_ENCRYPTION_KEY | aBNJhf2lPATLJqlm7iy-_uwWcVEeWycri9Y8qYhVav8= | 32-byte Fernet key

This is a real, valid Fernet key. If anyone uses this in their actual .env (common mistake — people copy sample values), every OAuth token stored in their database is decryptable with this key. HighTOKEN_ENCRYPTION_KEY used for two cryptographic purposes
# Used as Fernet key (symmetric encryption):
fernet = Fernet(settings.token_encryption_key.encode())

# Used as HMAC signing key (message authentication):
sig = hmac.new(settings.token_encryption_key.encode(), payload.encode(), hashlib.sha256)

One key, two purposes. Rotating the encryption key for OAuth tokens also breaks all outstanding confirmation tokens. Cryptographic principle: one key, one purpose.

If settings is ever passed to a logger, printed in a debug endpoint, or included in a traceback, your API keys are exposed in logs. Logs often go to CloudWatch where they're more widely accessible than your code.

MediumInconsistent placeholder style — %s in sync code suggests mixed DB drivers

    "SELECT status FROM approval_queue WHERE id = %s", (approval_id,)
).fetchone()

asyncpg uses $1; psycopg2 uses %s. Mixing these suggests the sync database layer is a different library from the async layer — a potential source of bugs if the wrong wrapper is used in the wrong context. MediumToken resource field can contain colons — breaks token parsing
# tokens.py
def issue_token(approval_id, action, resource):

    ...

If resource contains a colon (e.g., a Gmail thread ID like thread:18abc:subpart), rsplit(":", 4) extracts the wrong fields and verification fails. This is a logic bug — not SQL injection, but causes approval verification to silently fail or corrupt.

Code scan — CORS and Security Headers

HighNo CORS configuration visible in any file

main.py is not shown in IMPLEMENTATION.md, and no other file configures CORSMiddleware. FastAPI's default without CORS middleware allows no cross-origin requests — or if configured during development with allow_origins=["*"], allows all origins.

In development mode, it's common to temporarily set allow_origins=["*"] and forget to restrict it before shipping. The WebSocket URL wss://your-backend/ws/{userId} is a placeholder — confirming CORS hasn't been thought through for production.

Fix: Add explicit CORSMiddleware with the specific Electron and web origins listed.

HighNo security headers middleware

No X-Content-Type-Options, X-Frame-Options, Referrer-Policy, or Content-Security-Policy headers are configured. Without X-Frame-Options: DENY, your app can be embedded in an iframe on an attacker's site for clickjacking attacks. Without X-Content-Type-Options: nosniff, uploaded file attacks are easier.

Code scan — Rate Limiting

HighLLM endpoints have no rate limiting — cost explosion risk

The chat endpoint calls the Anthropic API (supervisor, categorizer batches, summarizer, style parser — potentially 5+ API calls per message). With no rate limit on /chat, a single malicious actor or runaway client can generate thousands of dollars of API cost in minutes. This is especially dangerous because your system has parallel worker execution.

MediumSend rate limit has a race condition — non-atomic check-then-act
# rate_limit.py — BROKEN
count = await db.fetchval("SELECT count(*) FROM audit_log WHERE ...")
return count < MAX_SENDS_PER_HOUR  # check happens here
# ... send happens here (outside this function)
# ... audit log written here (after send)
# Multiple concurrent requests can all pass the check before any are logged

50 concurrent requests can all pass count < 30 simultaneously, then all send, resulting in 50 emails when 30 was the intended maximum.

HighNo rate limiting on OAuth/auth endpoints

The auth flow endpoints have no rate limiting. An attacker can flood the OAuth callback endpoint with code exchange attempts or probe the approval endpoints. While individual items are time-limited, endpoint availability is not protected.
