Mail Agent — Complete Build Guide

This is the single source of truth for building Mail Agent end to end: a multi-agent system that reads, categorizes, summarizes, drafts, and sends email, and manages calendar events and reminders, all from natural-language instructions. Nothing has been built yet — this document covers the full original architecture, every gap found in it, and exactly what to do next.

How to read this guide. Each section carries a status badge. Resolved means the design is solid and ready to implement as-is. Gap filled means the original plan was missing this and it's been designed here for the first time. Needs decision flags a fork where you should make a call before building. The companion file IMPLEMENTATION.md has the actual runnable code for everything marked buildable.

1. Executive Summary

Mail Agent is a multi-agent AI system that lets a user manage their entire email and calendar workflow through natural-language instructions. Instead of clicking through inbox filters, the user types what they want — "summarize today's unread emails from clients and draft replies in a formal tone with my signature" — and a team of specialized AI agents plans the work, executes it through governed tools, and surfaces results in a control dashboard.

The system is built as a supervisor/worker multi-agent graph using LangGraph, with each worker agent owning one capability domain: reading, categorizing, drafting, sending, scheduling, reminders. Agents act through a strict tool layer with a permission system in front of every side-effecting action — send email, create event, modify calendar — so nothing irreversible happens without the right authorization level, which defaults to user confirmation.

Phase 1 ships against Gmail and Google Calendar. Phase 2 extends the same agent and tool abstractions to Microsoft Outlook / Graph API, reusing the entire orchestration layer — only the integration adapter changes.

1.1 What success looks like

  • A user can issue a single free-text instruction and have it correctly decomposed into the right sequence of agent actions.
  • Every categorization, summary, and draft is explainable — the user can see why an agent made a decision.
  • No email is sent and no calendar event is created without passing through the permission layer.
  • Formatting instructions (signature, tone, font, color, spacing) are honored precisely and consistently.
  • The architecture is provider-agnostic enough that adding Outlook in Phase 2 doesn't require rebuilding the agent layer.

2.2 Design principles

  1. Agent autonomy is opt-in, not default. Every agent starts conservative (draft + confirm); the user grants more autonomy per action type over time.
  2. Every action is auditable. Every tool call, decision, and state transition is logged with the reasoning that produced it.
  3. Tools are the only way agents touch the world. No agent calls the Gmail API directly from an LLM completion — it always goes through a typed, validated tool function.
  4. Formatting is a parameter, not a vibe. Style instructions (font, color, indentation, signature) are parsed into a structured style object, never left to free-text generation alone.
  5. Provider-agnostic core. The agent graph, permission system, and database schema are designed around abstract concepts — Message, Thread, Event, Contact — that both Gmail and Outlook adapters implement.

3.1 Request flow — happy path

  1. User types an instruction in the dashboard chat panel.
  2. Request hits the FastAPI backend, which loads conversation + user context and invokes the LangGraph supervisor.
  3. Supervisor agent decomposes the instruction into subtasks and routes each to the relevant worker agent.
  4. Worker agents call tools. Every side-effecting tool call is intercepted by the Permission Layer first.
  5. If the action requires approval, it's written to an approval queue and surfaced in the dashboard; execution pauses (LangGraph interrupt) until the user approves or rejects.
  6. Once approved (or pre-authorized), the tool executes against the Gmail/Calendar API.
  7. Results flow back through the supervisor, get logged to the audit trail, and stream to the dashboard UI.

⚠ Gap: no mention of where conversation history lives in this flow

The original flow assumes one instruction in, one response out. Real usage is multi-turn — "draft a reply to that one" referring to something three messages back. See § Conversation Memory for the fix: a conversations + messages table pair and how the supervisor loads recent turns into context before planning.

4. Multi-Agent Design with LangGraph Gap fixed

LangGraph models the system as a graph of nodes (agents/tools) and edges (control flow), with a shared state object passed between nodes. This gives parallel execution, conditional routing, and human-in-the-loop interrupts.

4.1 Why multi-agent instead of one big agent

  • Separation of concerns — a categorization agent and a drafting agent need very different prompts, context, and evaluation criteria. Mixing them in one mega-prompt degrades quality on both.
  • Parallelism — categorizing 50 unread emails and summarizing 5 threads can run concurrently as independent graph branches rather than serially.
  • Safety isolation — only agents that need send/create permissions are wired to those tools. The summarizer simply has no tool access to send_email — a structural guarantee, not a prompt instruction.
  • Easier evals — you can test "does the Categorizer assign correct labels" independently of "does the Drafter match tone."

4.2 Supervisor pattern

A Supervisor agent sits at the root of the graph. It doesn't execute domain work itself — it interprets the user's request, breaks it into a task list, and routes each task to the correct worker node.

⚠ Gap fixed: the original fan-in into one shared permission_gate node will corrupt state under real concurrency

The original design routes multiple parallel workers into a single permission_gate node without specifying how concurrent writes to the same state object are merged. In LangGraph, when two or more nodes run in parallel and write to the same state key, you must declare that key with a reducer via Annotated — otherwise the last writer silently overwrites the others' results, which is a real bug, not a theoretical one, once you have more than 2 parallel branches.

The fix: every list-shaped field in state that multiple agents append to (pending_approvals, completed_tasks, email_context, errors) must use operator.add as its reducer so LangGraph concatenates rather than overwrites. See the corrected state object below and the full working graph in IMPLEMENTATION.md → Section 3.

7. Permission & Approval System Resolved — this is the most important section

This is the most safety-critical subsystem. It determines whether an agent's proposed action executes immediately, waits for human approval, or is blocked outright.

7.1 Permission model

LevelBehaviorDefault For
AUTOExecutes immediately, no human step. Logged for audit.list_emails, get_thread, categorize, summarize
CONFIRMAction is prepared (e.g. draft created) and queued in dashboard; executes only after explicit user approval.send_email, create_event, update_event
BLOCKEDAgent may not propose this action at all under current rules.Bulk delete, forwarding to unknown external domains (configurable)

7.2 Approval queue & confirmation tokens

When a worker agent produces an action requiring CONFIRM, the Permission Layer writes a row to approval_queue, generates a single-use confirmation_token scoped to (user, action, resource, expiry), and pushes a real-time event to the dashboard via WebSocket. Graph execution interrupts at that node using LangGraph's interrupt() primitive.

Full implementation — including the corrected version that supports the multi-agent fan-out from Section 4 — is in IMPLEMENTATION.md → Section 5.

7.3 Dashboard approval UX

  • Pending approvals appear in a persistent "Needs your review" panel — never auto-dismissed.
  • Each approval shows the exact email/event content, the recipient(s), the agent's reasoning, and one-click Approve / Edit / Reject.
  • "Edit" lets the user modify the draft body or event details before approving — edits are saved back before the token is issued.
  • Users can set per-contact or per-label auto-approve rules, but send_email and create_event remain CONFIRM by default and require explicit opt-in to relax.

7.4 Audit logging

Every tool call — AUTO or CONFIRM — writes an immutable row to audit_log with timestamp, user_id, agent name, tool name, input parameters, output, the approval_id if applicable, and the LLM's stated reasoning. Non-negotiable for trust and debugging.

8a. Conversation Memory Gap fixed — was entirely missing

⚠ What was missing

The original spec's MailAgentState has a single instruction: str field — it models one request in, one response out, with no way to refer back to anything said earlier in the conversation. A real chat UI needs "draft a reply to that one" to resolve correctly, which requires actual conversation history feeding the Supervisor's planning step.

How it's resolved

  1. Every dashboard chat session maps to one row in conversations.
  2. Every user message and every agent-generated response is written to messages, in order, tagged with role (user / assistant / tool).
  3. Before the Supervisor plans a new instruction, the backend loads the last N turns (default: last 20, or last 4000 tokens, whichever is smaller) from messages and injects them into the LangGraph state's built-in messages field (inherited from MessagesState), so the Supervisor's prompt has real context.
  4. References like "that email" or "the one from Sarah" are resolved by the Supervisor against email_context entries already present in the recent conversation state — not by re-fetching from Gmail blindly.

Full loader function and the updated Supervisor prompt that uses it: IMPLEMENTATION.md → Section 6.

8b. Categorization Design Gap fixed — was underspecified

⚠ What was missing

The original spec says the Categorizer uses "rules + LLM judgment" but never defines the category taxonomy, where rules live, or what happens when a rule and the LLM disagree. That's enough ambiguity to stall implementation.

Resolved design

Default taxonomy (editable per user)

  • urgent — needs response within hours
  • action_needed — needs response, no hard deadline implied
  • meeting_request — contains scheduling intent
  • fyi — informational, no response expected
  • newsletter — bulk/marketing/subscription mail
  • personal — non-work correspondence

Resolution order

  1. Rules first. Check category_rules for a match (sender domain, exact sender, subject keyword, existing Gmail label). Rules are cheap and deterministic — evaluate before spending an LLM call.
  2. LLM fallback. If no rule matches, send the email metadata + snippet (not always the full body, to control cost — see § Cost) to a cheap/fast model call (Haiku-class) for classification into the taxonomy.
  3. Conflict handling. A matching rule always wins over LLM judgment for that message — rules represent explicit user intent and should never be silently overridden by the model. If the user wants the LLM to have final say, they delete or disable the rule.
  4. Confidence threshold. If the LLM call returns confidence below a threshold (modeled as a forced structured output with a confidence field), the email is tagged uncategorized rather than guessed, and surfaced for one-time user labeling — which becomes a new rule suggestion.

Schema for category_rules and the full classification function are in IMPLEMENTATION.md → Section 7.

9.2 New email trigger flow

  1. Worker polls Gmail history API (or uses Gmail Push Notifications via Pub/Sub) on a per-user schedule.
  2. New messages are written to email_cache.
  3. If the user has "auto-categorize new mail" enabled, a lightweight graph run triggers with only the Categorizer agent active.
  4. If a category matches a user-defined rule (e.g. "meeting requests → notify immediately"), a dashboard notification fires.
Note: Use Gmail Push Notifications (Pub/Sub) instead of polling once past prototype — polling burns API quota and adds latency. For the fast prototype, a 60-second poll loop with the history API is fine.

9.3 Why FastAPI + LangGraph, not one monolith prompt

  • FastAPI gives async request handling, WebSocket support, and OpenAPI docs for the dashboard to consume.
  • LangGraph gives durable execution — if the backend restarts mid-approval-wait, the checkpointer resumes the exact graph state rather than losing the user's in-flight request.
  • Decoupling orchestration from transport means you can later expose the same agent graph via CLI, Slack bot, or Outlook plugin without rewriting agent logic.

12. Phase 2: Microsoft Outlook / 365 Migration Plan Resolved

Because the architecture treats Gmail as one implementation of an abstract MailProvider interface, Phase 2 is primarily an adapter-writing exercise, not a redesign. Worker agents and tools call the abstract interface, never GmailProvider directly — the active provider resolves per-user based on which account they connected.

12.1 Outlook-specific notes

  • Auth: Microsoft Entra ID (Azure AD) app registration, OAuth 2.0 via MSAL, using Microsoft Graph API.
  • Scopes: Mail.Read, Mail.ReadWrite (drafts), Mail.Send, Calendars.ReadWrite.
  • Mail API: Microsoft Graph /me/messages, /me/mailFolders for categorization — Outlook uses folders + categories rather than Gmail labels, so map your internal Category model to Outlook's "categories" field.
  • Real-time sync: Microsoft Graph webhooks (subscriptions) instead of Gmail Pub/Sub push.
  • HTML rendering differences: Outlook's rendering engine (Word-based) has known CSS quirks — limited flexbox/grid support — so the formatting engine needs an Outlook-safe, table-based HTML template variant.

The abstract base classes (MailProvider, CalendarProvider) to build against from day one are in IMPLEMENTATION.md → Section 10 — write these even during Phase 1 so the Gmail adapter is the first implementation of an interface, not a hardcoded path.

13. Formatting & Style Engine Resolved

This directly addresses the requirement that the agent follow exact formatting commands — signature, font, color, indentation, spacing — reliably rather than approximately.

13.1 Structured style object

The Drafter agent never freehand-generates raw HTML styling per request. Natural-language formatting instructions are parsed into a structured StyleSpec, which is then deterministically rendered to HTML/CSS — this keeps formatting consistent and testable, instead of hoping the model gets it right each time.

style_spec.pyPython
class StyleSpec(BaseModel):
    tone: Literal["formal", "casual", "neutral", "match_sender"] = "neutral"
    font_family: str = "Arial"
    font_size_pt: int = 11
    text_color: str = "#000000"
    accent_color: str | None = None
    line_spacing: float = 1.15
    paragraph_indent_px: int = 0
    include_signature: bool = True
    signature_profile_id: str | None = None
    bullet_style: Literal["dash", "dot", "numbered"] = "dot"

13.2 Pipeline

  1. Free text ("make it formal, blue headers, add my signature, use Calibri") goes through a dedicated, lightweight LLM call constrained with forced tool-call output into the StyleSpec schema.
  2. Markdown body content from the Drafter + the StyleSpec are passed to a Jinja2 template renderer that produces final HTML email markup — the LLM never hand-writes closing HTML/CSS directly, which avoids inconsistent or broken markup.
  3. Signatures are stored as pre-approved HTML snippets per style_profiles row, never regenerated by the LLM — this guarantees the signature is always exactly what the user configured, with zero risk of the model altering it.
Note: Maintain two template variants — one standard HTML/CSS template, and one Outlook-safe table-based template for Phase 2.

The style-parsing prompt, the forced-schema tool call, and the full Jinja2 template (standard + Outlook-safe) are in IMPLEMENTATION.md → Section 11.

13a. Cost & Rate Limit Strategy Gap fixed — was never addressed

⚠ What was missing

The original spec never discusses what auto-categorizing every incoming email via an LLM call actually costs at volume, or how to stay inside Gmail/Calendar API quotas. This is the kind of gap that's invisible in a demo and painful in production.

Resolved approach

  • Rules before LLM calls. The category-rules resolution order in § Categorization Design already avoids paying for an LLM call whenever a deterministic rule matches — this should eliminate the majority of routine mail (newsletters, known senders) from ever hitting the model.
  • Cheap model for high-volume, simple classification. Use a Haiku-class model for categorization; reserve Sonnet-class for drafting, summarization, and supervisor planning, where reasoning quality actually matters.
  • Batch where possible. When categorizing a backlog (e.g. first sync of an inbox), batch multiple email snippets into a single prompt rather than one call per email, with the model returning a structured array of classifications.
  • Cache aggressively. A given thread's summary doesn't need regenerating unless the thread has new messages — store summaries with a thread_last_message_id watermark and only regenerate when it advances.
  • Gmail/Calendar API quotas. Gmail API has per-user and per-project quota units (e.g. a messages.list call costs quota units that scale with result size). Track usage centrally and apply exponential backoff on 429/403 quota errors rather than hard-failing.

The batched-categorization function and the quota-aware retry wrapper are in IMPLEMENTATION.md → Section 12.

13b. Error Handling & Partial Failure Gap fixed — was thin

⚠ What was missing

The original spec mentions idempotency testing as a goal but doesn't design for the actual failure modes: what happens when send_email succeeds at Gmail but the subsequent DB write fails? What happens when 1 of 50 emails times out mid-categorization batch?

Resolved approach

Send-then-record ordering

For any irreversible action, the provider-side call (the thing that can't be undone) always happens after the local intent is durably recorded, and the local record is updated after the provider confirms success — never the other way around:

  1. Write drafts row with status='approved' (durable intent, pre-send).
  2. Call gmail_client.send_draft().
  3. On success: update drafts.status='sent' and write the audit_log row in the same DB transaction.
  4. On failure: drafts.status stays 'approved' (not 'sent'), and the failure is logged. A reconciliation job can later check "approved but not sent" drafts against actual Gmail state (a sent message with matching provider_draft_id may still exist if only the DB write failed) before deciding whether to retry or alert the user — this avoids ever blindly re-sending.

Partial batch failure

When the Categorizer processes a batch and one item fails (timeout, malformed response), the failure is isolated to that item — it's written to errors in state and to audit_log, while the rest of the batch proceeds. The graph never aborts an entire multi-email operation because one email failed.

Idempotency keys

Every CONFIRM action's confirmation_token doubles as an idempotency key: if the same token is submitted twice (e.g. a double-click or a retried request), the Permission Service checks whether that token has already been consumed in approval_queue before allowing the tool to execute again.

The transactional send wrapper and the reconciliation job are in IMPLEMENTATION.md → Section 13.

⚠ Gap fixed: "labeled test set" needs an actual starting size

Start with a minimum of 50 hand-labeled emails per category for the Categorizer eval (300 total across the 6 default categories), and 20 hand-written instructions covering each agent combination for the Supervisor routing eval. This is a floor, not a target — but without a number, this step tends to get skipped entirely.

14.3 Integration & safety tests

  • Permission layer: automated tests asserting send_email and create_event cannot execute without a valid confirmation_token, including expired/reused token rejection.
  • Idempotency: re-running an interrupted graph after backend restart must not double-send an email or double-book a calendar slot.
  • Rate limiting / quota tests against Gmail and Calendar API limits, with backoff behavior verified.

15. Security & Compliance Resolved

  • Encrypt OAuth tokens at rest (AES-256 via a KMS-managed key) — never store raw access/refresh tokens in plaintext.
  • Minimize email body retention: cache only metadata/snippets by default; fetch full bodies on demand rather than persisting them, to reduce PII exposure at rest.
  • Scope requests to the minimum needed and clearly explain each scope's purpose on the consent screen.
  • All confirmation_tokens are single-use, short-lived (15-minute expiry recommended), and scoped to one specific action + resource — they cannot be replayed.
  • Implement per-user rate limits on agent-initiated sends to contain damage from a misconfigured rule or compromised session.
  • Log access to the audit_log table itself — who viewed whose audit history — since it contains sensitive behavioral data.
  • If targeting EU or regulated users, plan for GDPR data export/deletion endpoints early; the schema's user_id foreign keys with ON DELETE CASCADE make full account deletion straightforward.

15a. Bulk Campaigns — Personalized Outreach at Scale Extension — not yet built

This section covers extending Mail Agent from a personal email assistant into a system that can send personalized emails to a list of recipients — e.g. "send a personalized internship outreach email to these 200 recruiters, mentioning their company and role, in a professional but enthusiastic tone." This was not part of the original architecture. It's a real extension, not a relabeling of what already exists.

Honest scope note. The core multi-agent orchestration, drafting, and permission concepts in this product transfer directly to bulk sending — that part of the foundation is solid. But campaigns introduce genuinely new concerns this system doesn't have yet: a recipient dataset model, bulk approval (so the user isn't clicking "Approve" 200 times), and rate-limited sending (Gmail will throttle or flag an account that fires 200 sends in a tight loop). Treat this as new scope, not a free extension.

Draft generation, StyleSpec, signaturesTransfers directlyThe Drafter agent and style engine don't care if they're called once or 200 times.
Permission/approval conceptNeeds extensionCurrent model is one-action-one-approval. Needs a bulk approval mode — see §15a.4.
Recipient dataset handlingMissing entirelyNo concept of a recipient list, personalization variables, or campaign currently exists in the schema.
Rate-limited, queued sendingMissing entirelyCurrent send_email tool sends one message per gated call. Looping it 200 times will hit Gmail throttling and spam heuristics.
Campaign progress tracking, retriesMissing entirelyNo state for "176 sent, 8 failed, 16 queued."

15a.4 Bulk approval — the key UX/schema change

The existing approval_queue table is action-centric: one row, one pending action. Campaigns need a parent record representing "approve this batch of N personalized emails" with a sampled preview, not N individual rows requiring N clicks.

  • The dashboard shows a stratified preview — 3–5 actual generated drafts spanning different recipients, not just the first few — plus aggregate stats (recipient count, any personalization fields that came back empty or low-confidence).
  • Any individual draft where the personalization agent flagged missing or uncertain data (e.g. couldn't find a company name) is pulled out for individual review rather than silently bundled into "Approve All" — sending a wrong or blank personalization field to a real recipient is the actual risk here, and it's worth being deliberate about catching it before scale, not after.
  • "Approve All" issues one batch-scoped confirmation token that authorizes the Sender Queue to proceed through the whole batch — not N individual tokens.

15a.5 Rate-limited sending

Never loop send_email directly across a recipient list. Gmail enforces daily sending caps (lower on consumer accounts, higher on Workspace) and per-minute burst/anti-spam heuristics — firing 200 sends in a tight loop risks throttling or the account getting flagged, independent of the daily cap. The Sender Queue paces sends (e.g. 10/minute), checkpoints progress so a crash mid-campaign doesn't lose state or double-send, and retries failed sends in isolation without blocking the rest of the batch.
